# Authentication

## about_JiraPS_Authentication

# SHORT DESCRIPTION

JiraPS supports multiple authentication methods: HTTP basic authentication,
API tokens (Jira Cloud), Personal Access Tokens (Jira Data Center),
and session-based authentication.

# LONG DESCRIPTION

JiraPS supports the following authentication methods:

* HTTP basic authentication - username and password
* API tokens - for Jira Cloud (email + token)
* Personal Access Tokens (PAT) - for Jira Data Center (bearer token)
* Session-based authentication - creates a persistent session for subsequent commands

> Be sure to set JIRA up to use HTTPS with a valid SSL certificate if you are
> concerned about security!

## HTTP Basic

Each JiraPS function that queries a Jira instance provides
a `-Credential` parameter.
Simply pass your Jira credentials to this parameter.

```powershell
$cred = Get-Credential 'powershell'
Get-JiraIssue TEST-01 -Credential $cred
```

> HTTP basic authentication is not a secure form of authentication.
> It uses a Base64-encoded String of the format "username:password"
> and passes this string in clear text to Jira. Because decrypting this
> string and obtaining the username and password is trivial,
> the use of HTTPS is critical in any system that needs to remain secure.

## API Token (Jira Cloud)

API tokens are generated by the user for authenticating against Jira Cloud.

> As of December 1st 2018, Atlassian requires API authentication
> with Jira Cloud to always use API Tokens.
> More information in the [Deprecation notice](https://developer.atlassian.com/cloud/jira/platform/deprecation-notice-basic-auth-and-cookie-based-auth/).

JiraPS provides first-class support for API tokens via `New-JiraSession`:

```powershell
# Interactive prompt
$token = Read-Host -AsSecureString "Enter your API Token"
New-JiraSession -ApiToken $token -EmailAddress "you@example.com"
```

The API token and email are combined into a Basic authentication header
as required by Jira Cloud's API.

More information on API tokens and how to create one:
https://confluence.atlassian.com/cloud/api-tokens-938839638.html

## Personal Access Token (Jira Data Center)

Personal Access Tokens (PATs) are used for authenticating against
Jira Data Center (on-premise) installations.

JiraPS provides first-class support for PATs via `New-JiraSession`:

```powershell
# Interactive prompt
$pat = Read-Host -AsSecureString "Enter your PAT"
New-JiraSession -PersonalAccessToken $pat
```

The `-PersonalAccessToken` parameter also accepts the aliases `-PAT` and `-BearerToken`.
The PAT is sent as a Bearer token in the Authorization header.

More information on PATs for Data Center:
https://confluence.atlassian.com/enterprise/using-personal-access-tokens-1026032365.html

## Automation and CI/CD

For scripts, CI/CD pipelines, or other non-interactive scenarios,
you need to create SecureStrings programmatically instead of using `Read-Host`.

### From Environment Variables

The recommended approach for CI/CD - store secrets in environment variables:

```powershell
# Jira Cloud with API Token from environment variable
$token = ConvertTo-SecureString -String $env:JIRA_API_TOKEN -AsPlainText -Force
New-JiraSession -ApiToken $token -EmailAddress $env:JIRA_EMAIL

# Jira Data Center with PAT from environment variable
$pat = ConvertTo-SecureString -String $env:JIRA_PAT -AsPlainText -Force
New-JiraSession -PersonalAccessToken $pat
```

### From a String Variable

If you have the token in a variable (be careful with secrets in memory):

```powershell
$tokenPlainText = "your-api-token-here"
$token = ConvertTo-SecureString -String $tokenPlainText -AsPlainText -Force
New-JiraSession -PersonalAccessToken $token
```

### Creating Credentials Programmatically

For `-Credential` parameter with username/password:

```powershell
$username = "jira-user"
$password = ConvertTo-SecureString -String $env:JIRA_PASSWORD -AsPlainText -Force
$credential = [PSCredential]::new($username, $password)

New-JiraSession -Credential $credential
```

Security Note: Avoid hardcoding secrets in scripts. Use environment variables,
secret managers (Azure Key Vault, AWS Secrets Manager, HashiCorp Vault),
or CI/CD secret storage (GitHub Secrets, GitLab CI Variables).

## Jira Cloud vs. Data Center

JiraPS automatically detects whether your Jira instance is Cloud or
Data Center/Server by calling the `/serverInfo` endpoint (see
`Get-JiraServerInformation`). This detection is cached for the session.

On **Jira Cloud**, JiraPS adapts its behavior:

* Uses `accountId` instead of `username` for user identification
  (required since Atlassian's GDPR changes removed usernames from Cloud)
* Uses API v3 endpoints for JQL search (`/rest/api/3/search/jql`)
  with token-based pagination
* Warns when `New-JiraSession` is used, since cookie-based sessions
  are deprecated on Cloud — use `-Credential` with individual commands instead
* Handles HTTP 429 (rate limiting) with automatic retry and exponential backoff

No changes are required in your scripts. JiraPS selects the correct
API behavior based on the detected deployment type.

## Sessions

JiraPS supports persistent sessions that store authentication for subsequent commands.
Once a session is created, you do not need to pass credentials to each command.

### Creating a Session

Choose the method that matches your Jira deployment:

- Jira Cloud: Use `-ApiToken` with `-EmailAddress`
- Jira Data Center: Use `-PersonalAccessToken` (aliases: `-PAT`, `-BearerToken`)
- Either (legacy): Use `-Credential`

```powershell
# Jira Cloud
New-JiraSession -ApiToken $token -EmailAddress "you@example.com"

# Jira Data Center
New-JiraSession -PersonalAccessToken $pat

# Either
New-JiraSession -Credential $cred
```

Once you have created a session, you are done!
You do not need to specify credentials when running other commands - JiraPS will
manage this session internally.

The session is stored in the module's runtime.
This means that it will not be available in a new PowerShell session
or if the module is reloaded.

### Custom Authorization Headers

For advanced scenarios, you can also pass custom headers:

```powershell
$headers = @{ Authorization = "Bearer $token" }
New-JiraSession -Headers $headers
```

## What About OAuth 2.0 (3LO)

Jira Cloud supports OAuth 2.0 Three-Legged OAuth (3LO) for user-delegated access,
which is the recommended approach for integrations that act on behalf of Jira users.

JiraPS does not currently provide first-class OAuth 2.0 helpers. Tracking issue:
https://github.com/AtlassianPS/JiraPS/issues/101

For scripts and automation, use API tokens (Cloud) or Personal Access Tokens
(Data Center) as described above.
